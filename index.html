receipt-judge

智能报销票据提取与标注评估

项目概览

  • 面向 OA 报销场景，接收 PDF / PNG / JPG 票据。
  • 调用内网 Qwen 多模态模型，输出五大分类结构化 JSON （xcfy / zsfy / snjtje / qt / error）。
  • 提供 SSE 流式查看、历史回看、人工标注、 重复票据检测、统计分析。
  • 同时为 OA 提供异步提取接口，供后续入库回填。

技术栈：FastAPI · SQLAlchemy 2.0 async · React 19 · PostgreSQL · vLLM

本项目里"批处理"长什么样

① 标注队列 / 批量上传

  • 前端一次性扔进 N 张票据。
  • 浏览器侧维护 worker pool， 并发上传 + 轮询结果。
  • 后端 asyncio 协程池消费。

② 批量重推（rerun batch）

  • 选定某 receipt_type 下"错"的样本。
  • 整批重新走模型，性能数据落库。
  • 批次状态由后端自治收尾。

设计要点

为什么"后端自治"收尾

  • 前端可能关页面 / 断网，PATCH finish 不可靠。
  • 每条 session 写终态时主动调 try_finalize_batch。
  • SELECT ... FOR UPDATE 锁 batch 行，并发安全。
  • GET 接口扫 running 兜底，最终一致。

实时进度的取法

  • batch 表里的 success/failed 是收尾时一次性写。
  • 跑批途中通过对 sessions.status 的 GROUP BY 聚合得到 live_counts。
  • 索引 idx_verify_sessions_batch_status 让这条查询保持 O(group) 级。

Part 2

tts-bertvits2

BertVITS2 TTS · 在线请求批处理

项目概览

  • 基于 BertVITS2 的 TTS 服务，FastAPI + Uvicorn 部署。
  • 支持中 / 英 / 混合语言、多说话人、风格参考文本。
  • 瓶颈：单条请求 BERT 编码 + 声码器前向 占大头， 单卡难以同时跑很多条。
  • 关键工作：把"逐条 .infer"改写成 批量并行前向， 再针对两种调用场景给出两种调度。

TTSBus 的两层聚合

第一层：时间窗 / max_batch

  • 一个请求来了就开窗：time_window_ms=50。
  • 窗口期内继续收，凑满 max_batch=16 提前发车。
  • 超时也发车，保证延迟不退化。

第二层：_group_key 分组

  • 同一批内 speaker 可不同，但 language / sdp_ratio / noise_scale / style 必须一致才能合并 forward。
  • 不一致的拆成多个"车厢"，各跑一次 GPU。
  • speaker 不影响 batch 维度，仅作为 spk_id 喂入。
延迟 vs 吞吐：调 time_window_ms 大一点 → 吞吐 ↑、p99 ↑； 调小一点 → p99 ↓、吞吐 ↓。生产里按 SLA 反推。

性能实测：海光 DCU K100-AI 平台 软件验收

测试环境

  • 服务器：浪潮 5468H3
  • CPU：2× 海光 C86-4G 7490
  • 内存：1024 GB
  • 加速卡：8× 海光 K100-AI DCU
  • OS：Ubuntu 22.04
  • 测试日期：2025-11-18

结论摘要

  • CUDA 生态高度兼容：通过 torch 无缝迁移， 与英伟达侧持平。
  • 单卡 100 并发稳定跑出 7.1 RPS 的串行 /tts 吞吐。
  • 批处理接口（袋装）在并发 1、每批 50~100 条下 达到 5.6 RPS。
  • AES 加密通信、推理结果落盘、日志、SQLite 写入全部正常。

两个批处理的对比

以及为什么和课题相关

两种批处理的对比

receipt-judge · rerun batch tts-bertvits2 · TTSBus
请求来源 人工触发的成百上千条文件 线上 /tts 高并发流量
聚合时机 批次创建时就知道全集 动态时间窗聚合
并行单位 asyncio 协程 × 128 worker GPU batch 维度 × max_batch
瓶颈 远端 LLM 网关 RTT + token 数 本地 GPU forward 时长
收尾 事务 + 行锁，后端自治 Future.set_result，无需事务
失败兜底 GET 接口扫 running 兜底 逐条 Future.set_exception

共性 = 课题切入点

mindmap root((批处理共性)) 请求聚合 静态 (袋装 / rerun) 动态时间窗 (公交) 资源调度 worker pool 模型 max_concurrency 上限 group_key 分组前向 可观测 live_counts 实时聚合 逐阶段耗时 (prep/bert/fwd) tokens/s · items/s 容错 行锁 + lazy 收尾 逐条 Future 异常隔离 cancel 支持

与课题的关系

  • 课题方向：请求批处理 / 推理调度。
  • receipt-judge 提供"多请求 + 远端 LLM"侧的工程样本： 强调队列、并发、收尾一致性。
  • tts-bertvits2 提供"本地 GPU 模型"侧的样本： 强调时间窗聚合、按超参分组、跨请求特征缓存。
  • 两者拼在一起，覆盖了课题中"上层调度"和"底层并行前向"两层。
